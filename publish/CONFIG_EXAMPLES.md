# 🔧 Примеры конфигурации RDP Security Service

## 📍 Расположение файла

```
C:\ProgramData\RDPSecurityService\config.json
```

---

## 📖 Базовая структура

```json
{
  "port": 3389,
  "levels": [
    { "attempts": 3, "blockMinutes": 30 },
    { "attempts": 5, "blockMinutes": 180 },
    { "attempts": 7, "blockMinutes": 2880 }
  ]
}
```

### Параметры:

- **`port`** - Порт RDP для мониторинга (обычно 3389)
- **`levels`** - Уровни блокировки (массив правил)
  - **`attempts`** - Количество попыток подключения
  - **`blockMinutes`** - Время блокировки в минутах

---

## 🎯 Примеры конфигураций

### 1. Стандартный RDP порт (3389)

```json
{
  "port": 3389,
  "levels": [
    { "attempts": 3, "blockMinutes": 30 },
    { "attempts": 5, "blockMinutes": 180 },
    { "attempts": 7, "blockMinutes": 2880 }
  ]
}
```

**Логика:**
- 3 попытки → блокировка на 30 минут
- 5 попыток → блокировка на 3 часа (180 мин)
- 7+ попыток → блокировка на 2 дня (2880 мин)

---

### 2. Нестандартный порт (например, 3390)

```json
{
  "port": 3390,
  "levels": [
    { "attempts": 3, "blockMinutes": 30 },
    { "attempts": 5, "blockMinutes": 180 },
    { "attempts": 7, "blockMinutes": 2880 }
  ]
}
```

**Когда использовать:**
- RDP слушает на порту 3390 (изменено в реестре Windows)
- Используется порт forwarding
- Специфичная конфигурация сети

---

### 3. Агрессивная блокировка (для публичных серверов)

```json
{
  "port": 3389,
  "levels": [
    { "attempts": 2, "blockMinutes": 60 },
    { "attempts": 3, "blockMinutes": 1440 },
    { "attempts": 5, "blockMinutes": 43200 }
  ]
}
```

**Логика:**
- 2 попытки → 1 час
- 3 попытки → 1 день (1440 мин)
- 5+ попыток → 30 дней (43200 мин)

**Когда использовать:**
- Сервер доступен из интернета
- Высокий уровень атак
- Строгие требования безопасности

---

### 4. Мягкая блокировка (для внутренних сетей)

```json
{
  "port": 3389,
  "levels": [
    { "attempts": 5, "blockMinutes": 15 },
    { "attempts": 10, "blockMinutes": 60 },
    { "attempts": 15, "blockMinutes": 180 }
  ]
}
```

**Логика:**
- 5 попыток → 15 минут
- 10 попыток → 1 час
- 15+ попыток → 3 часа

**Когда использовать:**
- Внутренняя корпоративная сеть
- Пользователи часто забывают пароли
- Низкий риск внешних атак

---

### 5. Ультра-строгий режим (один шанс)

```json
{
  "port": 3389,
  "levels": [
    { "attempts": 1, "blockMinutes": 1440 },
    { "attempts": 2, "blockMinutes": 10080 },
    { "attempts": 3, "blockMinutes": 525600 }
  ]
}
```

**Логика:**
- 1 попытка → 1 день
- 2 попытки → 1 неделя (10080 мин)
- 3+ попытки → 1 год (525600 мин)

**⚠️ ВНИМАНИЕ:** Очень строгая политика! Используйте только если:
- Есть альтернативные методы доступа (консоль, IPMI)
- Знаете все легитимные IP адреса (добавьте в whitelist)
- Критичные системы с нулевой толерантностью к атакам

---

## 🔄 Применение изменений

После редактирования `config.json`:

### Вариант 1: Перезапуск сервиса
```batch
restart_service_admin.bat
```

### Вариант 2: Вручную
```batch
net stop RDPSecurityService
net start RDPSecurityService
```

### Вариант 3: PowerShell
```powershell
Restart-Service -Name RDPSecurityService -Force
```

## 📝 Рекомендации

### По времени блокировки:

| Длительность | Минуты | Применение |
|--------------|--------|------------|
| 15 минут     | 15     | Очень мягко (внутренняя сеть) |
| 30 минут     | 30     | Мягко (доверенная сеть) |
| 1 час        | 60     | Умеренно |
| 3 часа       | 180    | **Рекомендовано для большинства** |
| 1 день       | 1440   | Строго (второй уровень) |
| 2 дня        | 2880   | **Рекомендовано для публичных** |
| 1 неделя     | 10080  | Очень строго |
| 30 дней      | 43200  | Максимально строго |

### По количеству попыток:

| Попытки | Применение |
|---------|------------|
| 1-2     | Ультра-строгий режим (не рекомендуется) |
| 3       | **Рекомендовано для первого уровня** |
| 5-7     | **Рекомендовано для второго/третьего уровня** |
| 10+     | Мягкий режим (внутренняя сеть) |

---

## ⚠️ Важные замечания

1. **Whitelist всегда работает** - IP из whitelist никогда не блокируются
2. **Firewall правило обновляется автоматически** - при изменении block_list.log
3. **Конфиг не горячий** - требует перезапуска сервиса
4. **Порт должен совпадать** - с реальным портом RDP на сервере
5. **Валидация** - служба проверит корректность JSON при старте
